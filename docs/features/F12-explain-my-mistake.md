# F12 — Explain My Mistake

## 1. Purpose & Scope

After any exercise answered **incorrectly** (in practice or in a test review), the user can request an AI-generated explanation of their mistake. This is an explicit, on-demand AI touchpoint — never triggered automatically. The explanation states the correct answer, why it is correct, the rule in plain English, and a second Danish example of the same rule.

**Out of scope**:
- Automatic explanations (must be on demand)
- Translation answer *re-validation* (→ [F13](./F13-translation-answer-verification.md)) — a different touchpoint

---

## 2. Core Concepts & Requirements

### 2.1. Core Concepts

| Term | Definition |
|------|-----------|
| Mistake explanation | AI-generated breakdown of why an answer was wrong, with rule + extra example |
| On-demand | Generated only when the user explicitly requests it |

### 2.2. Requirements

#### 2.2.2. Endpoints

- `POST /exercises/:exerciseId/explainMistake` — request an AI explanation for a wrong answer.
  - Body: `{ userAnswer: string, cefrLevel: string }`.
  - Returns:
    ```json
    {
      "correctAnswer": "string",
      "explanation": "string",
      "rule": "string",
      "example": "string"
    }
    ```
    where `explanation` is why the correct answer is right (grammar rule or vocab note), `rule` states the rule simply in English, and `example` is a second Danish sentence demonstrating the same rule.

#### 2.2.4. Business Logic

- The delegate fetches the exercise (to get its content and correct answer) and the linked vocabulary item or grammar concept.
- It builds a prompt from those fields plus `userAnswer` and `cefrLevel`, and calls the AI service (see Section 6).
- No new persistent data model required; the explanation is returned in the response and not stored.
- Requesting an explanation does not change mastery or the exercise's correctness for the current session.

## 4. Constraints and Assumptions

- **Constraint** — On-demand only; this is a deliberate, opt-in exception to the no-live-AI rule.
- **Constraint** — AI call must be CEFR-aware.
- **Assumption** — Available both during practice (after a wrong answer) and in the post-test review (per incorrect item).

All open questions resolved.

| OQ-01 | Cache explanations per (exerciseId, userAnswer)? | **No caching.** The spec's constraint (no new persistent data model) stands. Add caching as a future optimisation only if AI cost becomes measurable. |

## 6. AI Integration

### 6.1. Provider

Explanations are generated by calling **Vertex AI directly** using `@google-cloud/vertexai`. The microservice runs in Cloud Run and relies on the ambient GCP service account credentials — no API key or extra auth setup required.

- **Package**: `@google-cloud/vertexai`
- **Model**: `gemini-2.5-flash-lite`
- **Project**: read from `GCP_PID` env var (standard across all Toto Cloud Run services)
- **Location**: read from `VERTEX_AI_LOCATION` env var (default: `europe-west1`)
- **Output**: `responseMimeType: 'application/json'` on the generation config, so the model returns structured JSON directly

This is the same pattern used in the `tome` app's `/api/llm-verify` backend route.

### 6.2. Prompt Contract

The prompt must include:
- The exercise prompt text and correct answer(s)
- The user's wrong answer
- The linked vocabulary item or grammar concept (name + explanation)
- The user's CEFR level

Instruction to the model: return a JSON object with fields `correctAnswer`, `explanation` (why the correct answer is right, tailored to the CEFR level), `rule` (the underlying grammar or vocabulary rule, stated simply in English), and `example` (a second Danish sentence demonstrating the same rule).

### 6.3. Testing

The Vertex AI call is wrapped behind an injectable client interface so that unit tests can supply a mock response without a live GCP call.

---

## 7. Technical Decisions

- **No caching** — consistent with the "no new persistent data model" constraint; revisit if AI cost becomes an issue.
- **No service-level rate limiting** — delegated to the infrastructure layer; this service remains stateless.
- **JSON output format** — the structured four-field response (`correctAnswer`, `explanation`, `rule`, `example`) matches the spec's stated output requirements and is cleaner to parse than free text.
- **Vertex AI direct call** — the microservice runs on Cloud Run in the same GCP project; ambient service account credentials make the call free of extra auth overhead. Follows the same pattern as `tome`'s `/api/llm-verify` route.
- **gemini-2.5-flash-lite** — lowest-latency, lowest-cost Gemini model available via Vertex AI; adequate for straightforward Danish grammar explanations. Upgrade to `gemini-2.5-pro` if explanation quality proves insufficient.
